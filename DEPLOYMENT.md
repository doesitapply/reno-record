# The Reno Record deployment guide

The recommended deployment path for The Reno Record is a **Render Web Service connected directly to the GitHub `main` branch**. Render is a good fit because this project is a long-running Node.js/Express service that serves both the API and the Vite-built React client from the bundled `dist/` output. Render supports automatic redeploys from a linked Git provider branch, so future edits from any editor will go live after they are committed and pushed to `main`.

## Recommended host

| Decision | Recommendation | Reason |
|---|---:|---|
| Primary host | Render Web Service | Supports long-running Node services, custom build/start commands, environment variables, custom domains, and GitHub auto-deploys. |
| Deployment trigger | Auto-deploy on push to `main` | Keeps the live site synchronized with repository changes from any editor. |
| Build command | `corepack enable && pnpm install --frozen-lockfile && pnpm build` | Installs the pinned pnpm toolchain and builds both the client and server bundle. |
| Start command | `pnpm start` | Runs `NODE_ENV=production node dist/index.js` from `package.json`. |
| Config file | `render.yaml` | Lets Render discover the service definition and prompt for required secrets. |

## Important portability warning

This application currently depends on Manus-specific storage and authentication variables. Before relying on an external Render deployment for production evidence intake, confirm that these endpoints are callable from Render, or replace them with external equivalents.

| Variable | Purpose | External-host risk |
|---|---|---|
| `BUILT_IN_FORGE_API_URL` | Manus Forge API endpoint used for object storage and presigned file access. | Uploads and evidence viewing may fail if the endpoint is only reachable from Manus-hosted infrastructure. |
| `BUILT_IN_FORGE_API_KEY` | Secret used by the storage proxy and upload flow. | Must be valid from Render and must never be committed to Git. |
| `OAUTH_SERVER_URL` | OAuth server used by the Manus auth SDK. | Admin login may fail if callback URLs or allowed origins are still tied to Manus hosting. |
| `VITE_APP_ID` | Public app identifier used by the client/auth flow. | Must match the deployed app configuration. |
| `OWNER_OPEN_ID` | Owner/admin identity used by the server. | Admin authorization depends on this value being correct. |
| `DATABASE_URL` | MySQL connection string used by Drizzle. | Must point to a database reachable from Render. |

## Render setup steps

1. Open [Render](https://render.com/) and create a new **Blueprint** or **Web Service** from the GitHub repository `doesitapply/reno-record`.
2. Select the `main` branch and enable automatic deploys.
3. If using the Blueprint flow, Render should detect `render.yaml` in the repository root. If using the manual Web Service flow, enter the build and start commands shown above.
4. Enter the required environment variables when Render prompts for values. `JWT_SECRET` can be generated by Render from the Blueprint configuration.
5. Deploy the service and verify the default Render URL first before adding a custom domain.
6. Test the public submission form, admin login, document upload, document viewing, and the **Open original** fallback button.
7. After the service is verified, add a custom domain or subdomain in Render and update the DNS records as instructed by the Render dashboard.

## Editing workflow after setup

Once the Render service is connected to GitHub, the normal publishing workflow is straightforward: edit the app in any editor, commit the change, push it to `main`, and let Render rebuild and redeploy automatically. If a change should not deploy, include Render's documented skip phrase in the commit message.

## Optional Railway fallback

Railway can also host a GitHub-connected Node service, but Render is the cleaner first choice for this repository because the `render.yaml` Blueprint can capture the required service configuration and secret prompts directly in source control. Railway remains a reasonable fallback if Render cannot reach the existing database, storage, or auth endpoints.
